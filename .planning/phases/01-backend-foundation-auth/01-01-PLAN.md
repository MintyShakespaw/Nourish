## Phase Goal

**As a** Nourish user, **I want to** create an account with email and password, **so that** my future food logs, weight, and workouts are stored in my account and retrievable from any device.

<objective>
This plan delivers the Walking Skeleton for the Nourish backend: a scaffolded Express + Drizzle + Docker-Postgres stack that boots from cold, applies one real migration, and serves a real `POST /api/auth/register` endpoint end-to-end. After this plan ships, a developer can `curl` the register endpoint and see a real PostgreSQL row.

Purpose: Lock the architectural decisions captured in `SKELETON.md` (directory layout, ORM patterns, error shape, env validation, token generation) before any other endpoint is built. Every subsequent endpoint reuses the same shape.

Output: Working backend with one endpoint (AUTH-01: register) backed by real DB writes. Subsequent plans (02 → schema, 03 → login/refresh/logout, 04 → password reset) build on this skeleton.

Mode: **Walking Skeleton vertical slice.** The plan is intentionally a thin end-to-end stack (Docker → migration → ORM → Express → controller → bcrypt → JWT → response), not a horizontal layer. After this plan, a real user can register.
</objective>

    import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core';

    export const users = pgTable('users', {
      id: uuid('id').primaryKey().defaultRandom(),
      email: text('email').unique().notNull(),
      passwordHash: text('password_hash').notNull(),
      createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
      updatedAt: timestamp('updated_at', { withTimezone: true }).defaultNow().notNull(),
    });

    export const refreshTokens = pgTable('refresh_tokens', {
      id: uuid('id').primaryKey().defaultRandom(),
      userId: uuid('user_id').references(() => users.id, { onDelete: 'cascade' }).notNull(),
      tokenHash: text('token_hash').notNull(),
      expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(),
      createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
      revokedAt: timestamp('revoked_at', { withTimezone: true }),
    });

    export const passwordResetTokens = pgTable('password_reset_tokens', {
      id: uuid('id').primaryKey().defaultRandom(),
      userId: uuid('user_id').references(() => users.id, { onDelete: 'cascade' }).notNull(),
      tokenHash: text('token_hash').notNull(),
      expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(),
      usedAt: timestamp('used_at', { withTimezone: true }),
      createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),

    **Step 3 — `backend/src/services/auth.service.ts` (Pattern 5):**
    ```typescript
    import jwt from 'jsonwebtoken';
    import bcrypt from 'bcryptjs';
    import crypto from 'node:crypto';
    import { env } from '../config/env.js';

    const ACCESS_TOKEN_EXPIRY = '15m';  // D-02
    const REFRESH_TOKEN_EXPIRY_DAYS = 30; // D-02

    export function generateAccessToken(userId: string): string {
      return jwt.sign(
        { sub: userId },
        env.JWT_ACCESS_SECRET,
        { expiresIn: ACCESS_TOKEN_EXPIRY, algorithm: 'HS256' }
      );
    }

    export function generateRefreshToken(): { raw: string; hash: string } {
      const raw = crypto.randomBytes(40).toString('hex');
      const hash = bcrypt.hashSync(raw, 10);
      return { raw, hash };
    }

    export function verifyAccessToken(token: string): { sub: string } {
      return jwt.verify(token, env.JWT_ACCESS_SECRET, { algorithms: ['HS256'] }) as { sub: string };
    }

    export function refreshTokenExpiry(): Date {
      return new Date(Date.now() + REFRESH_TOKEN_EXPIRY_DAYS * 24 * 60 * 60 * 1000);
    }
    ```

    import type { Request, Response } from 'express';
    import bcrypt from 'bcryptjs';
    import * as userRepo from '../repositories/user.repo.js';
    import * as tokenRepo from '../repositories/token.repo.js';
    import {
      generateAccessToken,
      generateRefreshToken,
      refreshTokenExpiry,
    } from '../services/auth.service.js';

    export async function register(req: Request, res: Response): Promise<void> {
      const { email, password } = req.body as { email: string; password: string };

      const existing = await userRepo.findByEmail(email);
      if (existing) {
        res.status(409).json({ error: 'Email already registered' });
        return;
      }

      const passwordHash = await bcrypt.hash(password, 12);
      const user = await userRepo.create({ email, passwordHash });

      const accessToken = generateAccessToken(user.id);
      const { raw: refreshToken, hash: refreshHash } = generateRefreshToken();
      await tokenRepo.createRefreshToken({
        userId: user.id,
        tokenHash: refreshHash,
        expiresAt: refreshTokenExpiry(),
      });

      res.status(201).json({ accessToken, refreshToken });
    }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile client → Express API | Untrusted JSON body crosses here; validated by Zod before reaching the controller |
| Express API → PostgreSQL | Drizzle parameterized queries — never string interpolation |
| Disk / DB backup | Anyone with DB read access must NOT be able to authenticate as a user — all passwords + refresh tokens stored as bcrypt hashes |
| Process env / .env file | Source of JWT signing secrets — Zod enforces `min(32)` at startup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | JWT signing | mitigate | `jwt.sign` and `jwt.verify` both pin `algorithm: 'HS256'` / `algorithms: ['HS256']`. Rejects `alg: none` and algorithm confusion attacks. Secret length enforced ≥32 chars by Zod at startup. |
| T-01-02 | Tampering | Drizzle queries on `users` table | mitigate | All queries use Drizzle's parameterized API (`eq`, `.values()`, `.where()`); never raw SQL string interpolation. SQL injection structurally impossible in the affected code paths. |
| T-01-03 | Information disclosure | `password_hash` column readable from DB | mitigate | bcrypt 12 rounds via `bcrypt.hash(password, 12)`. DB compromise yields hashes, not plaintext. Hashes are computationally expensive to crack. |
| T-01-04 | Elevation of privilege | `refresh_tokens.token_hash` column | mitigate | Store only `bcrypt.hashSync(raw, 10)`. Raw refresh token never persisted. DB compromise alone cannot impersonate users — attacker would also need the raw token from the user's device (Expo SecureStore, hardware-backed). |
| T-01-05 | Spoofing | Weak `.env` JWT secret | mitigate | Zod schema requires `JWT_ACCESS_SECRET.min(32)` and `JWT_REFRESH_SECRET.min(32)`. App refuses to start with weak secrets — fails fast before serving requests. |
| T-01-06 | Tampering | Helmet security headers | mitigate | `app.use(helmet())` in app.ts sets 15+ headers (HSTS, X-Frame-Options, etc.) for all responses. |
| T-01-07 | Information disclosure | CORS too permissive | mitigate | `cors({ origin: env.FRONTEND_URL })` restricts to `http://localhost:8081` (Expo dev) only. Production deployment will override via env. |
| T-01-08 | Information disclosure | Verbose error responses leaking internals | mitigate | `errorMiddleware` returns generic `{ error: 'Internal server error' }` for 500s; internals logged server-side via `console.error`, never sent to client. |
| T-01-09 | Denial of service | Login brute force | accept (in Plan 01) | Login endpoint does not exist yet in Plan 01. Rate limiting added in Plan 03 when `/api/auth/login` ships. Register endpoint accepts the risk in Plan 01 — bcrypt 12 rounds (~300ms per attempt) is a natural rate limiter against trivial flooding. |
| T-01-10 | Information disclosure | Email enumeration via register 409 | accept | A 409 reveals that the email is registered. This is standard for register endpoints (UX requires it). Mitigation for forgot-password (where enumeration matters more) ships in Plan 04 as a uniform 200 response. |

## Residual Risk

- Register endpoint can be flooded to brute-force email enumeration. Acceptable in Plan 01 — typical for register endpoints. If this becomes an issue, add rate limiting to `/api/auth/register` in a later phase.
- No CAPTCHA in MVP. Acceptable for a personal-use single-user app per project constraints.

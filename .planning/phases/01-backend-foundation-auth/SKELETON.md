# Walking Skeleton — Nourish Backend

This document records the architectural decisions made in Phase 1 that all subsequent phases will build on without renegotiation. The skeleton is the thinnest end-to-end stack proving Node runs, Docker runs, Drizzle connects, migrations apply, auth logic executes.

## End-to-End Slice

**Real user action proven:** A user calls `POST http://localhost:3000/api/auth/register` with email + password and receives `{ accessToken, refreshToken }`. The password is hashed via bcrypt, a row exists in the `users` table, and a hashed refresh token row exists in `refresh_tokens`.

**Proof path (in execution order):**
1. WSL Docker integration verified (`docker --version`)
2. `docker compose up -d` brings up `postgres:17-alpine` on `localhost:5432`
3. `npx drizzle-kit generate` produces `0000_initial_schema.sql` from TypeScript schema
4. `npx drizzle-kit migrate` applies migration to Postgres
5. `tsx watch src/server.ts` starts Express on port 3000 (env validated via Zod at startup)
6. `curl -X POST http://localhost:3000/api/auth/register -d '{"email":"test@test.com","password":"hunter2"}' -H 'Content-Type: application/json'` returns 201 with tokens
7. `psql` / Drizzle Studio shows `users` row with bcrypt hash and `refresh_tokens` row with bcrypt-hashed token

### Conventions

- **Primary keys:** `uuid('id').primaryKey().defaultRandom()` on every table — no serial/integer PKs.
- **Timestamps:** `timestamp({ withTimezone: true })` everywhere — Postgres UTC storage.
- **Foreign keys:** `references(() => parent.id, { onDelete: 'cascade' })` for user-owned data.
- **Idempotency:** All log-style tables (food_log, water_log, weight_log, workout_log) carry `idempotency_key text` + `UNIQUE(user_id, idempotency_key)`.
- **Denormalized snapshots:** `food_log` stores `calories`, `protein_g`, `carbs_g`, `fat_g` at log time — historical accuracy preserved even if `foods` values change.
- **Error response shape:** `{ error: 'message', details?: fieldErrors }` for 4xx/5xx; success returns flat object (auth) or `{ data: result }`.
- **Auth flow:** Bearer JWT in `Authorization` header; `requireAuth` middleware sets `res.locals.userId`. Never trust `req.body.userId`.
- **Token storage split:** Access (15min, HS256) → response body → Zustand memory on mobile. Refresh (30 days) raw → response body → Expo SecureStore on mobile; bcrypt hash → `refresh_tokens.token_hash` in DB.
- **Migration policy:** `npx drizzle-kit generate` to author, `npx drizzle-kit migrate` to apply — never `push`.

### Security Invariants (ASVS L1)

1. bcrypt 12 rounds for password hashes; bcrypt 10 rounds for refresh + reset token hashes (CPU vs. lookup tradeoff).
2. JWT always signed/verified with explicit `algorithm: 'HS256'` (reject `alg: none`).
3. `password_reset_tokens.used_at` set atomically with password update (single-use).
4. `/api/auth/forgot-password` always returns HTTP 200 (email enumeration prevention).
5. All request bodies validated by Zod; env validated by Zod at startup (`min(32)` on JWT secrets).
6. `express-rate-limit` on `/api/auth/login` (5 attempts / 15 min per IP).
7. `helmet()` and `cors({ origin: FRONTEND_URL })` mounted in `app.ts` before any routes.
8. Refresh tokens stored as bcrypt hash; raw token only ever in response body and mobile SecureStore.

### Schema Push Policy

| Operation | Command | When |
|-----------|---------|------|
| Author migration from schema diff | `npx drizzle-kit generate` | After any change in `src/db/schema/*` |
| Apply migration to DB | `npx drizzle-kit migrate` | After `generate`; in CI / dev / prod |
| Push without migration file | **PROHIBITED** | Never — bypasses the migration trail |

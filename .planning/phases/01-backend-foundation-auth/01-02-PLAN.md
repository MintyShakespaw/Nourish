This plan extends the schema laid in Plan 01 to cover every table Nourish will need across all five phases. By the end of this plan, the database has the full data model — users, profiles, foods, food/water/weight/workout logs, and a generic sync_outbox table — so that Plans 03/04 (more auth endpoints) and all later phases (Phase 2 logging UI, Phase 3 history, Phase 4 AI, Phase 5 offline flush) can write code against a stable schema.

## Residual Risk

- `food_log.idempotency_key` is nullable. A NULL key behaves as "no dedup" — same `(user_id, NULL)` is treated as distinct by Postgres UNIQUE semantics.
